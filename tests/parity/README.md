# Parity tests — Python ↔ TypeScript

This directory pins the TS port to the Python reference's verb output. Every change to a verb, the parser, the graph, or any threshold-bearing primitive is gated by these tests passing on **both** fixtures.

## Layout

```
tests/parity/
├── fixtures/
│   ├── sample-vault-14/              # 24 hand-written notes, copied from
│   │                                 # reference/examples/sample-vault/.
│   │                                 # Name kept "14" for test-stability;
│   │                                 # see SPEC.md §14.
│   └── test-vault-large/             # 200 deterministically-generated notes
│                                     # via scripts/generate-test-vault.ts
├── baseline/
│   ├── sample-14-brief.json          # full Brief (--section all)
│   ├── sample-14-buried.json         # --section buried-insight
│   ├── sample-14-connection.json
│   ├── sample-14-contradiction.json
│   ├── sample-14-thesis.json         # --section implicit-thesis
│   ├── sample-14-drift.json
│   ├── large-200-brief.json
│   └── large-200-{buried,connection,contradiction,thesis,drift}.json
├── ts.test.ts                        # vitest entry — lands in TASK-0.5
├── utils.ts                          # comparison helpers — TASK-0.5
└── README.md                         # this file
```

The fixtures are **inputs**; the baselines are **frozen outputs** of the Python CLI run against them. The TS test compares the TS engine's outputs against these baselines under the tolerances declared in `PRD.md` §8.1 and `SPEC.md` §0.

## Tolerances (PRD §8.1)

- Embedding similarity scores: `ε = 1e-5`.
- Set membership of returned findings: exact match.
- Ordering: exact match (ties broken deterministically — see SPEC.md §5.5, §6.7, §7.7).
- Citation paths: exact match.
- Quote extraction: exact match (byte-identical).

## Regenerating the baselines

You should rarely need to. The Python repo is **frozen** at tag `v0.0.11` per PRD §10 #3; the only legitimate triggers for regeneration are:

1. The `reference/` submodule pin moves (it shouldn't).
2. The fixtures themselves change — `sample-vault-14` copy refreshed from a newer `reference/examples/sample-vault/` (after a deliberate submodule pin bump), or `test-vault-large` regenerated against a deliberate seed/count change.
3. Embedding model intentionally rotated (default is `nomic-embed-text`).

When a regeneration is genuinely warranted, also write a `docs/decisions/` ADR explaining why and update `CHANGELOG.md`.

### Prerequisites

- **Python ≥ 3.13** with the reference package installed in a venv at `.venv-reference/`. Bootstrap once:
  ```bash
  uv venv .venv-reference --python 3.13
  ( cd reference && VIRTUAL_ENV="$(pwd)/../.venv-reference" uv pip install -e . )
  ```
- **Ollama** running on `http://localhost:11434` with the embedding model pulled:
  ```bash
  ollama serve &
  ollama pull nomic-embed-text
  ```

### Run

```bash
bash scripts/generate-baseline.sh
```

The script:

1. For each fixture, drops and rebuilds the SQLite index in `<fixture>/.basalt/basalt.db` (gitignored).
2. Calls `basalt index` to walk + parse + embed.
3. Calls `basalt brief --section <verb> --format json --top 3` for every verb plus `--section all`.
4. Writes the JSON to `tests/parity/baseline/<fixture-prefix>-<verb>.json`.
5. Sanity-checks each output is valid JSON.

### Useful overrides

| Variable | Default | Purpose |
| --- | --- | --- |
| `BASALT_BIN` | `.venv-reference/Scripts/basalt.exe` (Windows) or `.venv-reference/bin/basalt` | Path to the Python CLI. |
| `BASALT_TODAY` | `2026-05-09` | Locks Drift's window and Buried Insight's vault-aware thresholds to a stable "today". |
| `OLLAMA_URL` | `http://localhost:11434` | Where to find the Ollama daemon. |

### Synthetic vault — `test-vault-large`

The 200-note vault is generated by `scripts/generate-test-vault.ts`. It's deterministic given a seed:

```bash
bun run scripts/generate-test-vault.ts                    # default: seed=1556, 200 notes, today=2026-05-09
bun run scripts/generate-test-vault.ts --seed 42 --count 100
```

The two-runs-same-seed determinism property is exercised in CI; if a future PR changes the generator, run it twice to a `/tmp` location and `diff -r` to confirm.

The layout (rough — exact counts come from the generator's per-category caps):

| Folder | Purpose |
| --- | --- |
| `01-Daily/` | 45 daily notes, last ~9 weeks, with deliberate skewed project mentions (Drift signal). |
| `02-Projects/` | 4 projects (`Atlas`, `Beacon`, `Crucible`, `Driftwood`) with varying note counts. |
| `03-People/` | 10 people notes (short — exercise `MIN_WORD_COUNT` edges). |
| `04-Reading/` | 20 reading notes. |
| `05-References/` | 30 reference notes with internal wikilinks. |
| `06-MOCs/` | 6 MOC-shaped notes (high outgoing-link density — hub filter targets). |
| `07-Insights/` | Remainder — old, dormant prose notes (Buried Insight + Implicit Thesis fodder). |

## What the TS parity test does (TASK-0.5 forward)

`ts.test.ts` will:

1. Load each `baseline/<prefix>-<verb>.json`.
2. Run the same verb against the same fixture using `@basalt/core`.
3. Compare via `compareFindings` (in `utils.ts`):
   - Same set of `rel_path` keys.
   - Same ordering (after the verb's deterministic tie-break).
   - Similarity scores within `ε = 1e-5`.
   - Quotes exact-match.
4. Fail loudly with a side-by-side diff on the first divergence.

Failures don't always mean the TS port is wrong — they sometimes surface real Python bugs. When that happens, follow CLAUDE.md §6 (halt and ask), and record the resolution in `docs/parsing-decisions.md`.
